Good errors are easy to make, and hard to find.
* unit errors (degrees/radians, in my case) -- NASA can testify that these are [intergalactic-class troublemakers http://www.cnn.com/TECH/space/9911/10/orbiter.02/]
* messing with calculations on angles: when do you normalise them? where do you center that gaussian, how will it wrap around, since -Pi = Pi?
* having a const reference member:
  const PsName& _subscriberName
initialised with a field of a temporary object:
  _subscriberName(event->sender)
This is good because there's no warning at compilation (nor at runtime), and references look safer than pointers in general, but here they are not.
* not following the rules (see [ these pitfalls http://www.redhat.com/docs/manuals/enterprise/RHEL-3-Manual/cpp/macro-pitfalls.html] and [the FAQ http://www.faqs.org/faqs/C-faq/faq/], section 10) when writing a macro, and getting bitten badly, for instance
 #define ERROR(s) \
   cerr << "Error in function " << __FUNCTION__ << ": " << endl; \
   cerr << "  " << s << endl;
will fail miserably with this:
  if (! connect(toto))
      ERROR("cannot connect to toto");
The correct way to define a multiline macro is:
 #define ERROR(s) \
   { cerr << "Error in function " << __FUNCTION__ << ": " << endl; \
   cerr << "  " << (s) << endl; }
But be careful, because:
  if (! connect(toto))
      ERROR("cannot connect to toto");
  else 
      cout << "I am happy";
will not make you happy !
