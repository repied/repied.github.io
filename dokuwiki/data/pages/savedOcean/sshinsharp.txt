=== Cle SSH avec passphrase : version simple ====
Ces quatre commandes suffisent pour se connecter d'une machine de l'equipe a l'autre avec ssh, en utilisant un jeu de cles SSH. Cela permet de se connecter a ocean, qui n'accepte pas l'authentification par mot de passe (pour des raisons de securite).

 mkdir ~/.ssh
 cd ~/.ssh
 ssh-keygen -t rsa -f id_rsa
 cat id_rsa.pub >> authorized_keys

Pour tester que ca marche :
 ssh ocean

Pour se loger avec des clés sur un répertoire précis "mes_cles" différent de .ssh
 ssh -imes_cles/.ssh/id_rsa toto@xxxxx.dyndns.org

TRES IMPORTANT : n'utilisez jamais une passphrase vide ! Une passphrase doit etre compliquee et impossible a deviner. Si vous avez besoin d'automatiser des connexions, apprenez a utiliser ssh-agent+ssh-add, c'est tres simple.

NOTE: Les nouveaux utilisateurs (stagiaires...) doivent etre ajoutes dans un fichier de configuration d'ocean par un administrateur, pour pouvoir se connecter dessus (pour les autres machines, cela n'est pas necessaire).

=== Se connecter depuis l'exterieur avec ssh (version David) ===
Ces explications sont plus compliquées, mais elles offrent plus de securite que les precedentes : il est en effet plus raisonnable de garder la cle privee hors du home NFS.

Il faut savoir qu'ocean est la seule machine de l'équipe accessible de l'extérieur pour se loguer, et par SSH uniquement. Comme autre moyen de sortir du code, il y a le CVS (hébergé sur indigo), accessible de l'extérieur.
* la méthode David (R.), qui marche, qu'est-ce que vous croyez, pour se connecter de A vers B :
** générer une paire clé publique/privée, sur la machine A
  ssh-keygen -t rsa -f clef_pour_ocean
Ssh demande une passphrase, ie un mot de passe arbitrairement long et compliqué, qu'il faut retenir. Il faut faire ca sur un support sûr, une disquette ou une clef usb par exemple, mais pas sur un disque partage par NFS ! (tst tst, qui songerait à faire ça :)
** garder précieusement le fichier
  clef_pour_ocean
sur ledit support sûr, c'est notre clef privée, qui  va servir sur la machine A
** ajouter la clef publique 
  clef_pour_ocean.pub
dans 
  ~bellot/.ssh/authorized_keys 
(si on s'appelle bellot) sur la machine B (par exemple andrinople) ; si on est à l'extérieur on peut l'envoyer par mail à quelqu'un de l'équipe pour qu'il le fasse
** pour se loguer de chez soi (A) sur ocean (B), commencer par monter sa disquette (ou sa clé USB) avec la clé privée sur A, et faire
   ssh-add path/to/clef_pour_ocean   # (et taper sa passphrase)
   ssh bellot@ocean.inrialpes.fr     # toujours si notre login INRIA est bellot

=== Copier des fichiers avec scp ===
* Pour juste copier des fichiers de mon compte sur andrinople vers ma machine a l'exterieur (les ~ et * precedes d'un backslash(\) sont interpretes par ssh sur la machine distante, et non par le shell de la machine locale (extérieure a l'inria)) :
  # (sur un shell sur la machine a l'extérieur)
  scp lehy@ocean.inrialpes.fr:\~/.emacs .
  scp -r lehy@ocean.inrialpes.fr:/sharp/linux/games/ut /sandbox
  scp lehy@ocean.inrialpes.fr:/scratch/tmp/\* ~/tmp
=== Faire suivre des ports dans la connection ssh ===
Cédric a [une page plus détaillée sur la mise en place d'un tunnel http://bulle.inrialpes.fr/system/smtp-ssh-tunnel.html], et son utilisation pour envoyer des mails de l'extérieur du réseau en utilisant smtp-serv, sous Linux et Windows.

Ce truc est utile quand on veut communiquer entre une machine de l'extérieur (ou du réseau invité) et une machine de l'intérieur du réseau inria.

La commande ssh avec laquelle j'utilise imap-serv depuis chez moi :
 # a exécuter chez moi
 ssh -L10143:imap-serv.inrialpes.fr:143 ocean.inrialpes.fr watch date

Et après, quand je me connecte à ma machine de chez moi sur le port 10143, en fait ça parle à imap-serv sur le port imap (143).

Dans le même ordre d'idée,
 ssh -L8888:ocean.inrialpes.fr:80 ocean.inrialpes.fr watch date
me permet d'accéder au serveur web d'ocean si j'ouvre http://localhost:8888 dans Firefox chez moi.

et 
 # toujours à exécuter chez moi
 ssh -R 8889:localhost:22 ocean.inrialpes.fr watch date
me permet de me connecter chez moi (ma machine à la maison est derrière un firewall, je ne peux pas m'y connecter directement) à partir d'ocean comme ça :
 # a exécuter sur ocean
 ssh -p 8889 localhost

En fait, j'ai tout mis dans un script que je lance quand j'allume ma machine (à lancer sous X, avec ssh-agent installé et qui tourne):

 #!/bin/sh
 
 # add ssh key to ssh agent
 ssh-add -t0 < /dev/null
 
 echo "### Setting up link to INRIA ###"
 
 echo "Setting up tunnel:"
 echo "  arturo:8888 maps to ocean:80"
 echo "  ocean:8889  maps to arturo:22"
 while /bin/true; do
     ssh -L 8888:ocean.inrialpes.fr:80 -L10143:imap-serv.inrialpes.fr:143 -R 8889:localhost:22 ocean.inrialpes.fr watch date
     echo "got disconnected -- reconnecting"
 done
